2.3 Output
* A ranked list of feasible apartments ordered by total weekly commute time (ascending).
* The shortest path from each apartment to each destination (showing intermediate stations).
* Performance comparison between Dijkstra and Bellman-Ford.
2.4 Constraints
* Monthly rent <= $1,700
* Walking time to nearest transit station <= 15 minutes
2.5 Objective Function
* For each feasible apartment a, the total weekly commute time is:
* TotalCommute(a)  =  sum over all destinations d_i  of  [ shortest_time(a, d_i)  x  visits_per_week(d_i) ]
* where shortest_time(a, d_i) is the minimum travel time through the VTA transit graph from apartment a to destination d_i, computed via Dijkstra's algorithm. The apartment with the lowest TotalCommute is the optimal choice.

3. Algorithm Design
3.1 Approach
We model the problem as a shortest-path problem on a weighted undirected graph. Each apartment, destination, and VTA Light Rail station is a node. Edges represent traversable routes with non-negative time-based weights (in minutes). We apply Dijkstra's algorithm with a min-heap priority queue to compute shortest paths from each apartment to all destinations.


Dijkstra is chosen over Bellman-Ford because all edge weights (walking times, riding times) are non-negative, allowing the more efficient O((V + E) log V) complexity versus Bellman-Ford's O(V * E). The overall pipeline is:

Build Graph  ->  Filter by Constraints  ->  Run Dijkstra per Apartment  ->  Compute Weighted Score  ->  Rank & Output


3.2 Graph Structure
The graph contains 16 nodes and 25 edges:
* Node types: 5 apartment nodes (A1-A5), 3 destination nodes (D1-D3), 8 station nodes (S1-S8).
* Transit edges: 7 edges connecting adjacent Light Rail stations (weight = 3 min each).
* Walking edges: Connect each apartment/destination to nearby stations within 15-minute walking distance. Weight = Haversine distance * 1.3 / walking speed (5 km/h).
* Direct walking edges: If an apartment and destination are within 30-minute walking distance, a direct edge is added so Dijkstra can compare walking vs. transit.
3.3 Pseudocode
BUILD_GRAPH:
function BUILD_GRAPH(apartments, destinations, stations):
    G = empty adjacency list
    for each pair of adjacent stations (s_i, s_{i+1}):
        add edge (s_i, s_{i+1}, weight = 3 min) to G     // bidirectional
    for each apartment a:
        for each station s within 15-min walking distance:
            t = haversine_distance(a, s) * 1.3 / walk_speed
            add edge (a, s, weight = t) to G
    for each destination d:
        (same as apartment edges above)
    for each (apartment a, destination d) within 30-min walk:
        add direct walking edge (a, d, weight = walk_time)
    return G


DIJKSTRA (with min-heap):
function DIJKSTRA(G, source):
    dist = {node: infinity  for all nodes in G}
    prev = {node: null      for all nodes in G}
    dist[source] = 0
    heap = min-heap containing (0, source)
    visited = empty set
    while heap is not empty:
        (d, u) = extract-min from heap
        if u in visited: continue
        add u to visited
        for each neighbor v of u with edge weight w:
            if v in visited: continue
            if d + w < dist[v]:
                dist[v] = d + w
                prev[v] = u
                insert (dist[v], v) into heap
    return dist, prev


FIND_BEST_APARTMENT:
function FIND_BEST(G, apartments, destinations, budget, max_walk):
    results = []
    for each apartment a:
        if a.rent > budget: skip
        if walk_to_nearest_station(a) > max_walk: skip
        dist, prev = DIJKSTRA(G, source = a)
        score = 0
        for each destination d:
            score += dist[d] * d.weekly_visits
        append (a, score, paths) to results
    sort results by score ascending
    return results


3.4 Time and Space Complexity
Let V = total number of nodes, E = total number of edges, A = number of candidate apartments, D = number of destinations.
Component
	Time Complexity
	Space Complexity
	Dijkstra (single source)
	O((V + E) log V)
	O(V)
	Full pipeline (A apartments)
	O(A * (V + E) log V)
	O(V + E) + O(A * D)
	Bellman-Ford (comparison)
	O(V * E)
	O(V)
	

For our graph (V = 16, E = 25, A = 3 feasible apartments), the algorithm runs in under 0.1 milliseconds. Dijkstra is preferred over Bellman-Ford because all edge weights are non-negative (travel times cannot be negative).
